Executive Summary
SPECTRE is a next-generation, fully containerized, consent-first adaptive penetration testing platform. It accepts a target URL from a participant running a sandboxed application with a fake database, then autonomously executes a complete security assessment using a closed-loop AI feedback architecture — producing a real-time Extended Behaviour-Based Security Score (EBSS) backed by confirmed runtime evidence.

Core Innovation: The system adapts every attack decision in real time based on what the previous attack revealed — it learns during the test, not before it.

SPECTRE distinguishes itself from all existing tools across five dimensions:
	•	Explainable Random Forest ML (Model 1) with SHAP visualizations — not a black-box LLM
	•	NVD/CVE-grounded attack decisions (Model 2) — every attack traceable to a real CVE record
	•	Local Ollama LLM for platform fingerprinting — zero API cost, fully offline, conference-safe
	•	Full Kali Linux / Parrot OS tool library orchestrated by AI — not hardcoded scripts
	•	Live interactive dashboard + automated PDF report with EBSS score and remediation steps

The system operates exclusively in sandboxed, consented environments with fake databases, making it legally compliant, ethically sound, and fully reproducible — targeting conference demos and educational self-service use cases that existing autonomous agent systems cannot serve.

1. Problem Statement & Motivation
1.1 The Fundamental Gap
Current vulnerability assessment tools occupy two extremes: static scanners (Nmap, Nikto, ZAP) that run fixed rule sets without adaptation, and manual penetration testers who are expensive, unavailable at scale, and impossible to demo live at a conference. Between these extremes lies an unoccupied space — adaptive, explainable, lightweight automated penetration testing with real-time scoring. SPECTRE fills this space.

1.2 Limitations of CVSS-Only Scoring
	•	CVSS assigns identical scores regardless of whether an exploit actually works on a target system
	•	A CVSS 9.8 vulnerability may be completely unexploitable in a well-configured environment
	•	A CVSS 5.0 vulnerability may be trivially exploitable due to runtime misconfigurations
	•	No existing scoring system incorporates confirmed runtime exploit evidence into the score
	•	Security teams waste budget on CVSS-prioritized patching that doesn't reflect actual risk

2. Related Work & Honest Positioning
This section provides an honest comparison against the current state-of-the-art as of early 2026. Unlike a table designed to make SPECTRE win every column, we explicitly acknowledge where competing systems are stronger and articulate our differentiated value proposition.

2.1 xOffense (arXiv Sep 2025)
xOffense is a multi-agent LLM framework where specialized agents collaborate on offensive security tasks. Its raw attack reasoning capability exceeds SPECTRE's because it leverages large language models for tactical decision-making across a broader attack surface. However, it has no NVD/CPE grounding, no consent framework, requires external LLM API calls (cost + availability), lacks an EBSS scoring output, and is not deployable as a single Docker Compose command. It targets research offensive use cases, not educational self-service assessment.

2.2 ADAPT (2024–2025)
ADAPT uses a formal architecture-driven approach to automated penetration testing with knowledge graph representations of attack paths. It is well-structured academically but does not use ML feedback loops — attack paths are statically planned before execution begins. There is no live dashboard, no EBSS scoring, and no consent-first demo capability.

2.3 PentestGPT (2024–2026)
PentestGPT integrates LLM reasoning into penetration testing workflows and has an active open-source community. It has partial LLM-driven adaptation but lacks NVD/CVE grounding, SHAP explainability, and a quantified scoring output. Its primary mode is assisting human testers, not autonomous operation with a consent dashboard for self-service participants.

2.4 SPECTRE's Differentiated Position
We do not compete with xOffense on raw attack power. We beat all existing systems on deployability, explainability, ethical compliance, and quantified scoring.

3. System Architecture
3.1 Architectural Philosophy
SPECTRE is built on three architectural principles. Full decoupling: every service communicates exclusively through Redis Streams — nothing calls another service directly. Observability: every message in the system is visible through RedisInsight, making the AI's reasoning completely transparent. Reproducibility: the entire system launches with a single docker compose up command on any machine.

4.2 Local LLM Platform Classifier (Python + Ollama)
Instead of calling an external GenAI API (which introduces cost, latency, and reliability risk for conference demos), SPECTRE runs a local Ollama instance inside Docker using Llama-3.2-3B or Qwen2.5-7B. These models run comfortably on an 8GB laptop and perform platform classification at high accuracy. The model never generates attack payloads — only classifies platform stack from text observations.

  WHY LOCAL LLM IS THE RIGHT CALL:
  ▸ Ollama + Llama-3.2-3B runs on 8GB RAM inside Docker — no GPU required
  ▸ Zero API cost — critical for a conference where 50+ people use the tool
  ▸ Zero latency variance — consistent response time for live demo
  ▸ Offline capable — conference WiFi is notoriously unreliable
  ▸ No rate limits — all 50 simultaneous sessions get full LLM access
  ▸ Zero risk of the LLM refusing a platform classification request

4.6 ML Engine — Model 2: Attack Decision Engine (Python)
Model 2 is an NVD-informed expert decision tree that selects the next attack based on Model 1's vulnerability tags, the session's attack history, and CVE intelligence from the NVD service. It is deterministic and fully explainable — every decision can be traced to a specific rule, a specific CVE, and a specific observation.

  DECISION ENGINE — Core Logic
  ════════════════════════════════════════════════════════════════

  PRIORITY 1: KEV catalog matches (actively exploited in wild)
    if nvd.kev_match AND NOT already_tested(cve_id):
        → run targeted Nuclei CVE template immediately
        → priority: CRITICAL — test before anything else

  PRIORITY 2: Confirmed injectable endpoints
    if tags.sql_injectable AND NOT tags.waf_detected:
        → SQLMap --level=3 --risk=2 --batch
        → on success: SQLMap --dump --threads=4
        → on partial: try UNION extraction manually

    if tags.sql_injectable AND tags.waf_detected:
        → SQLMap --tamper=space2comment,between,charencode
        → try manual payload encoding variations
        → slow rate (1 req / 3s) to avoid block

  PRIORITY 3: Authentication surface
    if tags.auth_weak AND NOT tags.rate_limited:
        → Hydra HTTP-POST with rockyou-top-1000
        → on partial match: escalate to full rockyou-30k

    if tags.auth_weak AND tags.rate_limited:
        → slow brute (1 req / 8s, rotate User-Agent)
        → test for lockout bypass (X-Forwarded-For rotation)

  PRIORITY 4: Access control
    if tags.idor_present:
        → sequential ID scan 1–1000 on affected endpoint
        → UUID v1 timestamp prediction if UUID format detected
        → test horizontal privilege (swap own ID for others)

  PRIORITY 5: Verbose error exploitation
    if tags.verbose_errors AND nvd.db_match:
        → targeted Nuclei template for matched DB CVE
        → manual error-based extraction attempt

  FALLBACK: All techniques blocked
    → slow-rate attack (1 req / 10s)
    → rotate User-Agent pool (500 agents)
    → fragmented payload delivery
    → alternate HTTP method testing (PUT/PATCH/DELETE)
    → output: partial findings + WAF bypass recommendations

4.7 EBSS Scoring Engine (Python)
The Extended Behaviour-Based Security Score combines CVSS base metrics with confirmed runtime evidence. It updates progressively throughout the assessment — the dashboard shows the score climbing in real time as findings accumulate.

EBSS = min( CVSS_Base + (Behaviour_Weight × Behaviour_Severity) + (Exploit_Confirmed × 1.5) , 10.0 )

Progressive Scoring: After each attack round completes, the scoring engine recalculates EBSS and pushes an update to the ebss-updates Redis stream. The dashboard displays a live gauge rising from 0 to final score throughout the assessment. This is a key UX differentiator for conference demos.

	•	All experiments run 3 times per target, report mean ± standard deviation

11. Research Contribution Claims
11.1 Primary Contributions
The following four claims constitute the novel research contributions of SPECTRE. Each is verifiable through the evaluation metrics defined in Section 8.

	•	Closed-Loop Adaptive Attack Sequencing with Explainable ML
SPECTRE is the first system to combine Random Forest classification with SHAP explainability in a closed-loop penetration testing pipeline. Unlike LLM-based systems (black boxes), every attack decision can be traced to specific observable features. This explainability is verifiable, reproducible, and publishable.

	•	NVD/CVE-Grounded Attack Prioritization via CPE Matching
Every attack SPECTRE executes is semantically linked to a specific CVE record via CPE version matching, EPSS exploit probability, and KEV active exploitation status. No existing open-source adaptive pentest tool provides this three-signal CVE grounding.

	•	EBSS: A Runtime-Evidence-Augmented Vulnerability Score
EBSS extends CVSS with confirmed exploit evidence and ML-classified behaviour severity. The behaviour weight W is learned via regression on labeled outcomes — not an arbitrary constant. EBSS demonstrably correlates with real-world exploitability (EPSS) better than CVSS alone.

	•	Consent-First, Fully Reproducible, Single-Command Deployment
SPECTRE is the only adaptive penetration testing system designed as a consent-first conference demo tool. A single docker compose up command deploys the full system including local LLM, all attack tools, and live dashboard. This deployment model enables educational use cases that xOffense, PentestGPT, and ADAPT cannot serve.

11.2 Honest Limitations
Transparent acknowledgement of limitations strengthens a paper with reviewers. SPECTRE's current limitations are:
	•	Attack sophistication is limited vs. LLM multi-agent systems like xOffense — SPECTRE prioritizes explainability over raw reasoning depth
	•	Model 2 is a deterministic expert system, not a learned policy — Reinforcement Learning upgrade is a planned future direction
	•	v1 scope excludes binary/desktop software and direct database port exploitation
	•	Local LLM (3B params) may misclassify unusual or obfuscated tech stacks — external API fallback planned for v2

12. Legal & Ethical Framework
SPECTRE operates ONLY on sandboxed environments with fake databases, explicitly consented to by the participant.

12.1 Consent Architecture
	•	Five-item digital consent checklist — all items must be checked before assessment starts
	•	Consent record stored with session ID, timestamp, and participant details
	•	Scope is locked to the provided URL domain — no lateral movement to adjacent hosts
	•	Assessment auto-terminates at time limit or if participant clicks stop
	•	All session data (logs, raw responses) purged after PDF report is downloaded

12.2 Safe Operation Guarantees
	•	All attacks rate-limited to prevent genuine service disruption (configurable, default: standard)
	•	Fake DB design: participant seeds clearly labelled dummy data before assessment — no real PII at risk
	•	No real credentials, personal data, or sensitive information ever stored by SPECTRE
	•	All tool executions run inside Docker containers — zero access to host system
	•	Attack library excludes destructive payloads (DROP TABLE, rm -rf, etc.) — all attacks are read-only or reversible
	•	Emergency kill switch on dashboard terminates all attack subprocesses immediately
